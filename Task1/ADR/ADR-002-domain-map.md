# ADR-002: Карта доменов и границы сервисов (Domain Map)

## Статус

Принято

## Контекст

В монолитной архитектуре GoFuture уже существует деление на 9 доменных компонентов (Django Apps): Booking, Driver, Pricing, Payments, Payouts, Notification, Geography, Analytics и Fraud. Однако эти компоненты не являются независимыми:

- Все 9 доменов обращаются к **единой PostgreSQL** напрямыми SQL-запросами
- Booking Domain синхронно вызывает **6 других доменов**
- Нет инкапсуляции данных — домены свободно читают таблицы друг друга (например, `booking_domain → drivers`, `driver_domain → payouts`)
- Общая кодовая база (единый Django-проект) не позволяет деплоить домены независимо

Для перехода к микросервисам необходимо чётко определить Bounded Context каждого сервиса: его ответственность, владение данными, публичный контракт (API и события) и зависимости.

### Принцип организации

Согласно закону Конвея, структура системы повторяет структуру организации. В GoFuture есть **8 продуктовых команд**: Booking, Driver, Pricing, Payments, Notification, Geography, Analytics, Fraud. Карта сервисов строится в соответствии с этими командами, при этом Payouts выделяется в отдельный сервис (в рамках финансовой вертикали).

## Целевая карта сервисов (To-Be)

#### 📦 1. Booking Service

| Свойство              | Значение                                                                                                                                                    |
|-----------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Команда**           | Booking Team                                                                                                                                                |
| **Bounded Context**   | Управление жизненным циклом поездки                                                                                                                         |
| **Ответственность**   | Создание запроса на поездку, назначение водителя, управление статусами поездки (requested → accepted → in_progress → completed / cancelled), отмена и споры |
| **Ключевые сущности** | `Ride`, `RideRequest`, `RideStatus`, `RideHistory`                                                                                                          |
| **Собственная БД**    | PostgreSQL (ride_db)                                                                                                                                        |

**API (публичный контракт):**

| Метод | Endpoint                          | Описание                      |
|-------|-----------------------------------|-------------------------------|
| POST  | `/api/v1/rides`                   | Создать запрос на поездку     |
| GET   | `/api/v1/rides/{id}`              | Получить информацию о поездке |
| PUT   | `/api/v1/rides/{id}/status`       | Обновить статус поездки       |
| GET   | `/api/v1/rides?passenger_id={id}` | История поездок пассажира     |
| POST  | `/api/v1/rides/{id}/cancel`       | Отменить поездку              |

**Публикуемые события (Domain Events):**

| Событие         | Когда                 | Подписчики                                      |
|-----------------|-----------------------|-------------------------------------------------|
| `RideRequested` | Пассажир создал заказ | Pricing, Driver, Geography, Fraud               |
| `RideAccepted`  | Водитель принял заказ | Notification, Analytics                         |
| `RideStarted`   | Поездка началась      | Analytics, Geography                            |
| `RideCompleted` | Поездка завершена     | Payments, Analytics, Notification               |
| `RideCancelled` | Поездка отменена      | Payments (компенсация), Notification, Analytics |

**Синхронные зависимости (вызывает):**

| Сервис            | Зачем                             | Паттерн отказа                                  |
|-------------------|-----------------------------------|-------------------------------------------------|
| Pricing Service   | Получить цену при создании заказа | Circuit Breaker → показать «цена недоступна»    |
| Geography Service | Построить маршрут и оценить время | Circuit Breaker → использовать примерный расчёт |

**Примечание:** В текущем монолите Booking синхронно вызывает Driver, Payments, Fraud и Notification. В целевой архитектуре эти зависимости **переводятся на события**:
- **Driver** — подписывается на `RideRequested`, сам назначает водителя и публикует `DriverAssigned`
- **Payments** — подписывается на `RideCompleted`, сам инициирует платёж
- **Fraud** — подписывается на `RideRequested`, проверяет и публикует `FraudCheckPassed`/`FraudCheckFailed`
- **Notification** — подписывается на все бизнес-события, отправляет push

#### 📦 2. Driver Service

| Свойство              | Значение                                                                                                                                    |
|-----------------------|---------------------------------------------------------------------------------------------------------------------------------------------|
| **Команда**           | Driver Team                                                                                                                                 |
| **Bounded Context**   | Управление водителями и их доступностью                                                                                                     |
| **Ответственность**   | Профиль водителя, онлайн-статус, назначение на поездки, рейтинг, документы и верификация, текущее местоположение, «умное» перераспределение |
| **Ключевые сущности** | `Driver`, `DriverProfile`, `DriverStatus`, `DriverLocation`, `DriverRating`, `DriverDocuments`                                              |
| **Собственная БД**    | PostgreSQL (driver_db) + Redis (горячие данные: онлайн-статус, локация) + Elasticsearch (геопоиск доступных водителей)                      |

**API:**

| Метод | Endpoint                                         | Описание                                       |
|-------|--------------------------------------------------|------------------------------------------------|
| GET   | `/api/v1/drivers/{id}`                           | Профиль водителя                               |
| PUT   | `/api/v1/drivers/{id}/status`                    | Изменить статус (online/offline/busy)          |
| PUT   | `/api/v1/drivers/{id}/location`                  | Обновить местоположение                        |
| GET   | `/api/v1/drivers/nearby?lat={}&lng={}&radius={}` | Поиск ближайших доступных водителей            |
| POST  | `/api/v1/drivers/{id}/assign`                    | Назначить водителя на поездку (внутренний API) |

**Публикуемые события:**

| Событие                 | Когда                        | Подписчики                 |
|-------------------------|------------------------------|----------------------------|
| `DriverOnline`          | Водитель вышел на линию      | Analytics, Geography       |
| `DriverOffline`         | Водитель ушёл с линии        | Analytics                  |
| `DriverAssigned`        | Водитель назначен на поездку | Booking, Notification      |
| `DriverLocationUpdated` | Обновление GPS-координат     | Geography (кеш), Analytics |

**Потребляемые события:**

| Событие         | От кого | Действие                                        |
|-----------------|---------|-------------------------------------------------|
| `RideRequested` | Booking | Запустить алгоритм поиска и назначения водителя |
| `RideCompleted` | Booking | Освободить водителя, обновить рейтинг           |
| `RideCancelled` | Booking | Освободить водителя                             |

**Примечание:** Elasticsearch для геопоиска остаётся внутри Driver Service, а не в Geography — потому что **Driver владеет данными о местоположении водителей**. Geography отвечает за маршруты и карты, а не за водителей.

#### 📦 3. Pricing Service

| Свойство              | Значение                                                                                                                                       |
|-----------------------|------------------------------------------------------------------------------------------------------------------------------------------------|
| **Команда**           | Pricing Team                                                                                                                                   |
| **Bounded Context**   | Ценообразование                                                                                                                                |
| **Ответственность**   | Расчёт стоимости поездки, динамическое ценообразование (surge pricing), ценовые правила по регионам, расчёт заработка водителя, тарифные планы |
| **Ключевые сущности** | `PriceQuote`, `PricingRule`, `SurgeMultiplier`, `Tariff`, `DriverEarnings`                                                                     |
| **Собственная БД**    | PostgreSQL (pricing_db) + Redis (кеш ценовых правил и surge-коэффициентов)                                                                     |

**API:**

| Метод | Endpoint                               | Описание                                                    |
|-------|----------------------------------------|-------------------------------------------------------------|
| POST  | `/api/v1/pricing/quote`                | Рассчитать стоимость поездки (input: маршрут, тариф, время) |
| GET   | `/api/v1/pricing/surge?zone_id={}`     | Текущий surge-коэффициент для зоны                          |
| POST  | `/api/v1/pricing/earnings`             | Рассчитать заработок водителя за поездку                    |
| GET   | `/api/v1/pricing/tariffs?region_id={}` | Тарифы региона                                              |

**Публикуемые события:**

| Событие           | Когда                       | Подписчики                                         |
|-------------------|-----------------------------|----------------------------------------------------|
| `PriceCalculated` | Цена рассчитана для заказа  | Booking (callback), Analytics                      |
| `SurgeUpdated`    | Изменился surge-коэффициент | Analytics, Notification (информирование водителей) |

**Зависимости:** Pricing — **«лист» в графе зависимостей**. Он никого не вызывает синхронно, только отвечает на запросы. Это делает его хорошим кандидатом для раннего выделения.

|--------------------|----------|------------------------------|

| Событие            | Когда                                | Подписчики                                                    |

### Сводная таблица владения данными (Data Ownership)

Ключевой принцип: **каждая таблица/сущность принадлежит ровно одному сервису**. Остальные получают данные через API или события.

| Сущность (таблица)                               | Владелец сервис  | Кто раньше читал напрямую           | Как получают данные в To-Be |
|--------------------------------------------------|------------------|-------------------------------------|-----------------------------|
| `rides`, `ride_requests`                         | **Booking**      | Booking, Analytics, Fraud           | API / Events                |
| `drivers`, `driver_profiles`, `driver_locations` | **Driver**       | Booking, Driver, Geography, Payouts | API / Events                |
| `pricing_rules`, `tariffs`, `surge`              | **Pricing**      | Pricing, Booking                    | Sync API                    |
| `payments`, `transactions`, `refunds`            | **Payments**     | Payments, Fraud, Analytics          | API / Events                |
| `payouts`, `driver_balances`                     | **Payouts**      | Payouts, Driver                     | API / Events                |
| `notifications`, `device_tokens`                 | **Notification** | Notification                        | Internal only               |
| `routes`, `zones`, `hot_zones`                   | **Geography**    | Geography, Booking                  | Sync API                    |
| `analytics_events`, `reports`                    | **Analytics**    | Analytics (все таблицы!)            | Event streaming only        |
| `fraud_checks`, `blacklists`, `risk_scores`      | **Fraud**        | Fraud, Payments                     | Sync API / Events           |

## Альтернативы

1. **Объединить Payments и Payouts в один сервис «Finance»** — отвергнуто. Это два разных денежных потока (in vs out) с разными SLA, разными внешними интеграциями (Яндекс Пэй vs API Банка) и разными регуляторными требованиями (PCI DSS только для Payments). Разделение даёт возможность масштабировать и деплоить их независимо.

2. **Оставить Elasticsearch геопоиска водителей в Geography** — отвергнуто. Водитель и его местоположение — это данные Driver Service. Geography отвечает за карты и маршруты (внешний API Яндекс Карт), а не за поиск водителей. Смешение ответственности создавало бы циклическую зависимость.

3. **Выделить отдельный «User Service»** для профилей пассажиров — отвергнуто на текущем этапе. Профиль пассажира пока не является отдельным Bounded Context: он используется через JWT-токен (basic info) и не имеет сложной бизнес-логики. Можно выделить позже при необходимости.

4. **Сделать Fraud частью Payments** — отвергнуто. Fraud проверяет не только платежи, но и поездки (`RideRequested`). Объединение ограничило бы его scope и нарушило SRP.

## Компромиссы

- **9 сервисов — это много для старта**. Реально выделяться они будут поэтапно (см. ADR-003). На переходный период часть доменов остаётся в монолите.
- **Sync API для Pricing и Fraud** (от Booking и Payments) — это осознанный выбор, потому что цена и проверка на фрод нужны *до* продолжения бизнес-процесса. Компромисс: латенси увеличивается (сетевой вызов вместо in-process), но зато изоляция отказов через Circuit Breaker.
- **Elasticsearch внутри Driver** увеличивает инфраструктурную сложность этого сервиса, но обеспечивает чистую инкапсуляцию данных.
- **Analytics потребляет события из всех сервисов** — широкий fan-in. Нужен надёжный Event Bus (Kafka) и schema registry для контроля формата событий.
