# ADR-005: План бесшовной миграции данных

## Статус
Принято

## Контекст
В монолитной архитектуре GoFuture все домены используют одну общую БД PostgreSQL (shared database). База данных насчитывает миллионы записей (`rides`, `drivers`), обслуживая 500 тыс. поездок в день. Остановка системы для миграции (Maintenance Window) строго запрещена бизнес-требованиями. Требуется выделить данные для каждого нового микросервиса в его собственную базу (Database-per-Service), обеспечив нулевой даунтайм и возможность надежного отката (Rollback).

## Решение

В качестве основной стратегии извлечения данных мы принимаем паттерн **Change Data Capture (CDC)** на базе инструментов **Debezium** и **Apache Kafka**.

Мы **отвергаем** использование "Двойной записи" (Dual Writes) непосредственно из кода приложения, так как он несовместим с высокими NFR (500K транзакций) из-за проблем с распределенной консистентностью и риском сетевых сбоев.

### 1. Механизм CDC (Change Data Capture)
- **Debezium Connector** напрямую подключается к транзакционному журналу старой БД (Write-Ahead Log, WAL в PostgreSQL).
- Любое изменение стейта базы (INSERT, UPDATE, DELETE) мгновенно, надежно и асинхронно превращается в Kafka-событие.
- Sink Connector (или Consumer в коде нового сервиса) читает эти события из шины и применяет их в новую базу.

### 2. План миграции (Поэтапный перенос из монолита)
Для переезда таблиц каждого отдельного микросервиса мы будем применять **6-фазный цикл**:

1. **Подготовка (Setup):** Поднимается изолированная база данных микросервиса (напр., `driver_db`).
2. **Снапшот (Initial Sync):** Debezium делает первичный консистентный срез исторического массива данных из PostgreSQL монолита и "заливает" в целевую базу.
3. **Репликация (Incremental Sync):** Debezium начинает непрерывно стримить дельты из WAL. Монолит является источником истины (master). Новая БД полностью идентична старой с задержкой в доли секунды.
4. **Переключение Чтения (Switch Read):** 
   - Клиентский Read-трафик переключается на новый микросервис.
   - Если данные отображаются с ошибками, мы просто откидываем рубильник Gateway назад на монолит (никакого риска, т.к. "master"-ом записи остается монолит). 
5. **Переключение Записи (Switch Write & Reverse CDC):**
   - Запись и функции "мастера" переносятся в микросервис, весь трафик теперь проходит через новую БД.
   - Выполняется **разворот CDC (Reverse CDC)**. Если в монолите еще жив скрипт, который зависит старой таблицы, мы настраиваем Debezium считывать изменения теперь уже *из новой БД* и писать транзакции *обратно в монолит*. Старая таблица в монолите становится read-only копией для легаси-кода.
6. **Очистка (Cleanup):** Как только остальной монолит перестает дергать старую таблицу (переписан на вызовы API/события), Reverse CDC глушится, а таблица легаси безжалостно удаляется (Truncate/Drop).

### 3. Гарантии и стратегия отката (Rollback)
- **Откат на фазах 1–4:** Трафик возвращается на API монолита одной кнопкой в Nginx Gateway. Данные не утрачены. Никакого расхождения стейта нет.
- **Откат на фазе 5:** Поскольку у нас настроен **Reverse CDC**, старая БД в монолите зеркально содержит в себе все свежие изменения (даже те, что вносились уже новым микросервисом). В случае критического сбоя, запись плавно возвращается на монолит, и ни одна транзакция за время работы микросервиса не будет потеряна.

## Альтернативы
1. **Двойная запись из приложения (Dual Writes)** – Отвергнуто. Требует либо 2PC координатора, либо распределенную сагу. При сбое записи в новую базу (которая еще может быть нестабильна) приложение будет "возвращать" 500-й ответ клиентам, снижая аптайм монолита, либо приведет базы к неконсистентному виду (Split Brain).
2. **Дамп через Nightly Maintenance Window** – Отвергнуто. Агрегатор такси не спит ночью. Плюс это подход Big-Bang, который нарушает требование об инкрементальном переезде.

## Компромиссы
- Поднятие Debezium, настройка коннекторов и монитинга лага репликации (Replication Lag) добавляет **нагрузку на инфраструктурную команду**. Это сложная distributed система, за которой нужно наблюдать.
- Модель Kafka (at-least-once) требует от микросервисов строгого соблюдения **идемпотентности**. Апдейты в базу данных при перечитывании (upsert) должны давать тот же результат стейта без спама дубликатами.
