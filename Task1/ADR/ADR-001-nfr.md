# ADR-001: Нефункциональные требования к целевой микросервисной архитектуре GoFuture

## Статус

Принято

## Контекст

Монолитная платформа GoFuture — такси-агрегатор, работающий на рынках Юго-Восточной Азии и Южной Америки. Текущая архитектура (единая Django-кодовая база + единая PostgreSQL) не справляется с ростом:

- Пиковые нагрузки свыше 100 тысяч активных поездок приводят к каскадным отказам
- MTTR превышает 4 часа
- Сборка монолита занимает более 30 минут
- Разработка новых функций затягивается на полгода+
- Невозможно масштабировать отдельные домены независимо

Для перехода на микросервисную архитектуру необходимо определить чёткие, измеримые нефункциональные требования (NFR), которые станут основой всех архитектурных решений.

## Решение

### 1. Performance (Производительность)

| Сервис               | Метрика                    | Целевое значение               | Обоснование                                                                     |
|----------------------|----------------------------|--------------------------------|---------------------------------------------------------------------------------|
| Booking Service      | Latency создания поездки   | p95 ≤ 300ms, p99 ≤ 500ms       | Критический пользовательский сценарий — пассажир ждёт ответа в реальном времени |
| Booking Service      | Throughput                 | ≥ 500 000 конкурентных поездок | Прямое бизнес-требование                                                        |
| Pricing Service      | Latency расчёта цены       | p95 ≤ 100ms, p99 ≤ 200ms       | Вызывается синхронно из Booking при каждом заказе, на critical path             |
| Driver Service       | Latency поиска водителя    | p95 ≤ 200ms, p99 ≤ 400ms       | На critical path — пассажир ждёт назначения водителя                            |
| Geography Service    | Latency геопоиска/маршрута | p95 ≤ 150ms, p99 ≤ 300ms       | Зависит от внешнего API (Яндекс Карты), включает сетевой латенси                |
| Payments Service     | Latency инициации платежа  | p95 ≤ 500ms, p99 ≤ 1000ms      | Зависит от внешнего платёжного шлюза, допустим более высокий латенси            |
| Notification Service | Время доставки push        | ≤ 5 секунд (best effort)       | Некритичный путь, асинхронная доставка                                          |
| Analytics Service    | Задержка обработки событий | ≤ 5 минут (near real-time)     | Аналитика не на критическом пути, допустима задержка                            |
| Fraud Service        | Latency проверки           | p95 ≤ 100ms, p99 ≤ 200ms       | На critical path Booking и Payments                                             |

### 2. Availability (Доступность)

| Сервис               | Целевой SLA | Допустимый даунтайм/месяц | Обоснование                                             |
|----------------------|-------------|---------------------------|---------------------------------------------------------|
| Booking Service      | 99.95%      | ~22 мин                   | Ядро бизнеса — без заказов нет дохода                   |
| Pricing Service      | 99.95%      | ~22 мин                   | Без цены нельзя создать заказ                           |
| Payments Service     | 99.95%      | ~22 мин                   | Без оплаты поездка не завершается                       |
| Driver Service       | 99.95%      | ~22 мин                   | Без водителей нет поездок                               |
| Geography Service    | 99.9%       | ~44 мин                   | Может деградировать gracefully (кешированные маршруты)  |
| Fraud Service        | 99.9%       | ~44 мин                   | При недоступности — fallback: пропускать с логированием |
| Notification Service | 99.5%       | ~3.6 часа                 | Потеря/задержка push-уведомления не блокирует бизнес    |
| Analytics Service    | 99.0%       | ~7.2 часа                 | Аналитика не влияет на пользовательский опыт            |
| Payouts Service      | 99.9%       | ~44 мин                   | Выплаты обрабатываются батчами, допустима задержка      |

**Целевой MTTR для всей платформы: ≤ 30 минут** (сейчас > 4 часов).

### 3. Scalability (Масштабируемость)

| Требование                     | Описание                                                                                   | Целевые сервисы                                |
|--------------------------------|--------------------------------------------------------------------------------------------|------------------------------------------------|
| Горизонтальное масштабирование | Каждый сервис масштабируется независимо, добавлением реплик                                | Все сервисы                                    |
| Auto-scaling                   | Автоматическое масштабирование при росте нагрузки (триггер: CPU > 70% или RPS > threshold) | Booking, Pricing, Driver, Geography            |
| Мультирегиональность           | Развёртывание в нескольких географических регионах одновременно                            | Все сервисы (data residency для Payments)      |
| Независимый деплой             | Каждый сервис деплоится независимо, без координации с другими командами                    | Все сервисы                                    |
| Масштабирование данных         | Шардирование/партиционирование данных для сервисов с большими объёмами                     | Booking (история поездок), Analytics (события) |

### 4. Reliability (Надёжность)

| Требование           | Описание                                                                           | Паттерн                                |
|----------------------|------------------------------------------------------------------------------------|----------------------------------------|
| Изоляция отказов     | Сбой одного сервиса не вызывает каскадного отказа остальных                        | Circuit Breaker (Hystrix/Resilience4j) |
| Graceful degradation | При недоступности некритичного сервиса система продолжает работать с ограничениями | Fallback-стратегии, Feature Flags      |
| Retry с backoff      | Повторные попытки при временных сбоях с экспоненциальной задержкой                 | Exponential Backoff + Jitter           |
| Idempotency          | Повторная обработка запроса даёт тот же результат                                  | Idempotency keys (Payments, Payouts)   |
| Timeouts             | Каждый вызов между сервисами имеет таймаут                                         | Timeout budget: 80% от SLA latency     |
| Bulkhead             | Изоляция пулов соединений/потоков между сервисами                                  | Thread pool isolation                  |

### 5. Data Consistency (Согласованность данных)

| Поток данных       | Модель консистентности     | Механизм             | Обоснование                                                                                                     |
|--------------------|----------------------------|----------------------|-----------------------------------------------------------------------------------------------------------------|
| Booking → Payment  | Eventual Consistency       | Saga (Orchestration) | Платёж может быть разнесён во времени, но должен быть гарантирован. Saga обеспечивает компенсирующие транзакции |
| Booking → Driver   | Eventual Consistency       | Domain Events        | Назначение водителя — асинхронный процесс с подтверждением                                                      |
| Booking → Pricing  | Strong (синхронный запрос) | Sync API call        | Цена должна быть актуальна на момент создания заказа                                                            |
| Payments → Payouts | Eventual Consistency       | Domain Events        | Баланс водителя обновляется после подтверждения платежа                                                         |
| Все → Analytics    | Eventual Consistency       | Event Streaming      | Аналитика допускает задержку до 5 минут                                                                         |
| Payments → Fraud   | Strong (синхронный запрос) | Sync API call        | Проверка на мошенничество обязательна до проведения платежа                                                     |

### 6. Security (Безопасность)

| Требование         | Описание                                                                    | Целевые сервисы                     |
|--------------------|-----------------------------------------------------------------------------|-------------------------------------|
| PCI DSS compliance | Изоляция платёжных данных в отдельном сегменте сети с ограниченным доступом | Payments Service                    |
| Data Residency     | Персональные данные хранятся в регионе пользователя (PDPA, LGPD)            | Все сервисы с персональными данными |
| mTLS               | Шифрование трафика между сервисами (mutual TLS)                             | Все inter-service коммуникации      |
| AuthN/AuthZ        | Централизованная аутентификация (JWT), авторизация на уровне сервиса (RBAC) | API Gateway + каждый сервис         |
| Secrets Management | Централизованное хранение секретов (Vault)                                  | Все сервисы                         |
| Audit Logging      | Логирование всех изменений финансовых данных                                | Payments, Payouts                   |

### 7. Observability (Наблюдаемость)

| Требование          | Инструмент                  | Описание                                                         |
|---------------------|-----------------------------|------------------------------------------------------------------|
| Distributed Tracing | Jaeger / OpenTelemetry      | Сквозная трассировка запросов между сервисами с correlation ID   |
| Метрики             | Prometheus + Grafana        | RED-метрики (Rate, Errors, Duration) для каждого сервиса         |
| Логирование         | Loki / ELK                  | Структурированные логи в JSON с correlation ID                   |
| Alerting            | Alertmanager + PagerDuty    | SLA-based alerting: алерт при приближении к бюджету ошибок       |
| Health Checks       | Liveness + Readiness probes | Каждый сервис реализует `/health/live` и `/health/ready`         |
| SLO Dashboards      | Grafana                     | Дашборды с error budget burn rate для каждого критичного сервиса |

### 8. Deployability (Развёртываемость)

| Требование                  | Целевое значение               | Текущее значение         |
|-----------------------------|--------------------------------|--------------------------|
| Время сборки одного сервиса | ≤ 5 минут                      | > 30 минут (монолит)     |
| Частота деплоев             | Несколько раз в день на сервис | Раз в 2 недели (монолит) |
| Время выхода новой функции  | ≤ 2 недели                     | ≥ 6 месяцев              |
| Rollback time               | ≤ 5 минут                      | > 30 минут               |
| Blast radius деплоя         | Один сервис (одна команда)     | Вся платформа            |

| Сервис           | Latency (p99)  | SLA    | Auto-scale | Consistency          | Приоритет   |
|------------------|----------------|--------|------------|----------------------|-------------|
| **Booking**      | ≤ 500ms        | 99.95% | ✅          | Saga (orchestrator)  | 🔴 Critical |
| **Pricing**      | ≤ 200ms        | 99.95% | ✅          | Strong (sync)        | 🔴 Critical |
| **Driver**       | ≤ 400ms        | 99.95% | ✅          | Eventual (events)    | 🔴 Critical |
| **Payments**     | ≤ 1000ms       | 99.95% | ❌          | Saga                 | 🔴 Critical |
| **Geography**    | ≤ 300ms        | 99.9%  | ✅          | Eventual             | 🟡 High     |
| **Fraud**        | ≤ 200ms        | 99.9%  | ❌          | Strong (sync)        | 🟡 High     |
| **Payouts**      | ≤ 2000ms       | 99.9%  | ❌          | Eventual (events)    | 🟡 High     |
| **Notification** | best effort    | 99.5%  | ❌          | Eventual (events)    | 🟢 Medium   |
| **Analytics**    | near real-time | 99.0%  | ❌          | Eventual (streaming) | 🟢 Medium   |

## Альтернативы

1. **Единые NFR для всех сервисов** — отвергнуто. Избыточно: зачем тратить ресурсы на 99.95% SLA для Analytics, если он не влияет на пользовательский опыт?
2. **Более жёсткие SLA (99.99%)** — отвергнуто. Требует значительно больше инвестиций в инфраструктуру (мультирегиональная active-active репликация), на текущем этапе избыточно.
3. **Strong consistency везде** — отвергнуто. Распределённые транзакции (2PC) резко снижают throughput и увеличивают latency. Eventual consistency с гарантией доставки — достаточна для большинства потоков.

## Компромиссы

- **Eventual consistency** между сервисами означает, что данные могут быть временно рассогласованы (например, баланс водителя обновится с задержкой). Это приемлемо для бизнеса, но требует корректной обработки edge cases.
- **Разные SLA** усложняют мониторинг — нужны разные SLO-дашборды и alerting policies для каждого уровня.
- **PCI DSS** для Payments Service увеличивает стоимость инфраструктуры (отдельный сегмент сети, аудит), но это регуляторное требование.
- **Мультирегиональность** требует решения проблемы data residency — персональные данные не должны покидать регион.
