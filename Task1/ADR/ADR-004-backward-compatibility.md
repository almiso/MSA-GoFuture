# ADR-004: Обеспечение обратной совместимости при миграции

## Статус
Принято

## Контекст
Процесс миграции монолита GoFuture займет около 12 месяцев. В течение этого времени монолит и новые микросервисы будут работать параллельно. Мобильные клиенты, внешние B2B-шлюзы и различные компоненты системы не могут быть обновлены одновременно. 

Поэтому необходимо утвердить стандарты и паттерны, которые будут гарантировать плавный переезд без нарушения работы существующих механизмов (Zero Downtime) и провала в бизнес-метриках (500K поездок).

## Решение

Мы принимаем комплексную стратегию обратной совместимости на 4-х уровнях: Сеть (Маршрутизация), API Контракты, События и Данные.

### 1. Маршрутизация и паттерн Strangler Fig
- **API Gateway (Nginx):** Nginx выступает единым фронтом. Мобильные клиенты вообще не знают о микросервисах — они по-прежнему отправляют запросы на один домен.
- **Canary Deployment (Канареечный деплой):** При переключении готового переписанного модуля трафик пускается дозировано: **1% → 10% → 50% → 100%**. В Nginx или Ingress контроллере настраиваются правила распределения весов.
- Если на этапе 10% возникает деградация (растет доля 500-х ответов по метрикам Prometheus), фиче-флаг или настройки Gateway автоматически перенаправляют 100% трафика обратно на монолит.

### 2. Версионирование API
- Используем **URI Versioning** (например, `/api/v1/rides`). Это наглядно, просто кэшируется в CDN и легко маршрутизируется в Nginx.
- Если новый микросервис выпускает `v2` (ломающее изменение), он обязан либо временно поддерживать `v1` (транслируя вызовы внутри), либо Nginx должен делать проброс, пока клиенты не перейдут. Удалять `v1` можно только тогда, когда аналитика API покажет 0 пакетов по этому маршруту в течение 30 дней.

### 3. API Контракты (Contract Testing)
- Переход от тестирования продюсеров к **Consumer-Driven Contracts (CDC)** с использованием **Pact**.
- Потребитель (например, сервис Notification) описывает, *какие* поля он ожидает от Booking Service, и эти ожидания фиксируются в виде контракта (Pact Broker).
- Никакой деплой продюсера не пройдет пайплайн CI/CD, если он нарушает зафиксированные ожидания какого-либо потребителя.

### 4. Толерантность и События в Kafka
- Внедрение **Schema Registry** (Apache Avro или Protobuf) для всех внутренних Event-Bus сообщений (`RideRequested`, `PaymentCompleted`). Это защищает от публикации событий в ошибочном формате, который "крашнет" консьюмеров.
- **Паттерн Tolerant Reader**: Весь код потребителей (будь то монолит или микросервис) пишется так, чтобы **игнорировать неизвестные поля**. Это позволяет продюсеру безболезненно расширять полезную нагрузку (добавлять новые ключи в JSON/события) без необходимости договариваться и обновлять код подписанных сервисов.

### 5. Эволюция Схемы БД (Expand and Contract)
Любые обратно несовместимые изменения как в базе монолита, так и в новых микросервисах проводятся по 3-ступенчатой схеме:
1. **Expand (Расширение):** Создается новая колонка / таблица. Старая не удаляется. Приложение начинает *одновременно* писать в обе, но все процессы (выборки) использует старую, чтобы не сломать "медленных" консьюмеров.
2. **Migrate (Миграция):** Исторические данные переводятся в новую колонку фоновым джобом. Код переписывается на чтение из нового места. Выкатывается релиз.
3. **Contract (Сжатие):** После успешной обкатки в production, старая колонка или таблица окончательно удаляется (Drop).

## Альтернативы
1. **Версионирование API через Headers (`Accept: application/vnd.gofuture.v1+json`)** — Отвергнуто. Увеличивает когнитивную нагрузку, осложняет диагностику через cURL/браузер, усложняет роутинг в Nginx.
2. **Построение шлюза в самом монолите (In-App Proxy)** — Отвергнуто. Маршрутизацией должен заниматься инфраструктурный слой (Nginx), чтобы при полном отказе монолита микросервисы продолжили работу независимо.

## Компромиссы
- Подход Consumer-Driven Contracts (Pact) требует дополнительных затрат на написание тестов и развертывание Pact Broker. Первое время это замедлит команды, но вдолгую спасет от катастрофических багов при деплоях.
- Стратегия Expand and Contract создает временный мусор в БД (дублирующиеся колонки), что требует высокой дисциплины от команды — не забывать проводить фазу Contract и зачищать легаси.
