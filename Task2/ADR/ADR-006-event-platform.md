# ADR-006: Событийная платформа и обработка потоков

## Статус
Принято

## Контекст
Платформа GoFuture должна выдерживать 500k+ конкурентных активных поездок, что подразумевает генерацию нескольких тысяч событий в секунду (телеметрия водителей, статусы поездок, перерасчеты цен). Для реализации Event-Driven Architecture (EDA) требуется выбрать брокер сообщений, формат топиков, правила партиционирования, а также инструменты для надежной публикации и stream-обработки.

## Решение

### 1. Брокер сообщений: Apache Kafka
Выбран как индустриальный стандарт для EDA благодаря высокой пропускной способности, возможности повторного чтения (replay) и гарантии персистентности на диске.

### 2. Схемы Топиков и Партиционирование
Топики создаются по правилу: `[domain].[entity].[event_type].[version]`
Партиционирование критически важно для сохранения строгой последовательности (ordering) концептуально связанных событий.

| Топик                          | Partition Key | Обоснование                                                                                                                                         |
|--------------------------------|---------------|-----------------------------------------------------------------------------------------------------------------------------------------------------|
| `booking.ride.lifecycle.v1`    | `ride_id`     | Гарантирует строгий порядок статусов одной поездки (Requested -> Assigned -> Completed). Критично для корректной работы паттерна Saga координатора. |
| `driver.telemetry.location.v1` | `driver_id`   | Порядок GPS-координат для конкретного водителя (отрисовка трека). Это высоковолатильный топик.                                                      |
| `pricing.surge.updated.v1`     | `region_id`   | Зонированный мультипликатор цены. Ожидаются консьюмеры (Карты, Analytics), которым важен хронологический паттерн удорожания по району.              |

Для сериализации событий используется **Schema Registry (Protobuf/Avro)**, гарантирующая строгую типизацию и backward/forward совместимость сообщений.

### 3. Инструмент Stream-процессинга: Apache Flink
Динамическое ценообразование (Surge Pricing) требует агрегации данных из разных топиков (спрос = `booking.ride`, предложение = `driver.telemetry`) в реальном времени.
- Наши сервисы пишутся на Go и Python, поэтому KStreams (привязанный к JVM/Java) неудобен для встраивания.
- Выбираем **Apache Flink** в качестве независимого выделенного Data Processing кластера.
- **Окна времени (Tumbling Windows):** Flink-джоба группирует события за промежутки в 2 минуты, сопоставляя их по `region_id`, вычисляет индекс спроса (Demand/Supply ratio), генерирует новый коэффициент и тут же публикует его в топик `pricing.surge.updated.v1`.

### 4. Гарантии доставки: Transactional Outbox
Для решения проблемы двойной записи (dual writes failure), когда мы успешно сохранили данные заказа в свою PostgreSQL, но упали за миллисекунду до отправки события в Kafka:
- Подтверждаем паттерн **Transactional Outbox**.
- В рамках *одной SQL-транзакции* Booking Service создает строку в таблице `rides` и бизнес-событие в таблице `outbox_events`.
- Изолированный **Debezium CDC** (выбранный в Задании 1) выступает надежным транспортом — он асинхронно читает базу и со 100% гарантией переносит `outbox` в Kafka (At-Least-Once).

### 5. Потребители: Idempotency & DLQ
- Из-за механизма "At-Least-Once", брокер может доставить событие повторно при retry. Все сервисы-потребители обязаны быть **Idempotent Consumers** (использовать `event_id` как дедупликатор в своей БД).
- Сообщения, бросающие фатальные ошибки валидации при обработке консьюмером, автоматически сбрасываются в **Dead Letter Queue (DLQ)** топик для разбора, чтобы не блокировать обработку всей партиции (Poison Pill prevention).

## Компромиссы
- **Сложность инфраструктуры:** Kafka + Schema Registry + Flink + Debezium (ZooKeeper/KRaft) требуют высококвалифицированных Platform и Data-инженеров для круглосуточной эксплуатации кластеров.
- **Риск "Горячих партиций" (Hot Partitions):** Если партиционировать Surge по регионам, центральный регион мегаполиса может перегрузить конкретную ноду (Broker). Потребуется fine-tuning партицирования.
