# ADR-008: Стратегия мониторинга событийной платформы

## Статус
Принято

## Контекст
При переходе на распределенную Event-Driven архитектуру (Kafka, Flink, CDC, 9 микросервисов) классический мониторинг исключительно "по логам и HTTP-статусам" перестает быть эффективным. Если микросервис А опубликовал событие, а консьюмер Б упал, HTTP 500 нигде не появится, и бизнес начнет терять деньги скрытно. 
Для платформы с нагрузкой в 500k+ поездок требуется проактивный `Observability` стек с сильным фокусом на асинхронные метрики.

## Решение

Мы внедряем единый стандарт **Cloud-Native Observability**, состоящий из Трех Столпов (Three Pillars): Metrics, Logging, Tracing.

### 1. Архитектурный стек инструментов (Tools Selection)
- **Сбор метрик (Metrics): [Prometheus]** 
  Все микросервисы (Go Prometheus client, Python prometheus-client), базы данных, и инфраструктура (Kafka JMX Exporter, Flink Metrics Reporter) отдают числовые метрики в едином формате (Pull model).
- **Визуализация и Алертинг (Dashboards & Alerts): [Grafana]** 
  Единая панель управления. Использование Grafana Alertmanager для пороговых push-уведомлений в дежурные каналы.
- **Распределенная трассировка (Tracing): [Jaeger]** (или Grafana Tempo) 
  Критически важно для EDA. `trace_id` из API Gateway прозрачно и насквозь пробрасывается в заголовках (Headers) Kafka-сообщений. Это позволяет на одном графе визуально "склеить" весь путь транзакции: `Запрос клиента -> Booking -> Kafka -> Driver -> Kafka -> Payments` и найти точное место торможения.
- **Логирование:** **ELK/EFK стек (Elasticsearch, Fluentd, Kibana)**.

---

### 2. Ключевые метрики (Golden Signals) событийной платформы

Для обеспечения надежности (Reliability) Kafka, Stream Processing (Flink) и микросервисов мы обязаны мониторить специфические показатели. Ниже приведен список утвержденных метрик.

#### Метрики брокера (Kafka Broker & Zookeeper/KRaft)
1. **Under-Replicated Partitions (URP):** Метрика здоровья кластера (#1 приоритет). Должна быть на нуле. Если растет, значит брокеры (Brokers) падают или сеть деградирует.
2. **Bytes In / Bytes Out (Throughput):** Аномальные спады по трафику укажут на отказ API Gateway или крупных продюсеров.

#### Метрики потребителей (Consumers / Microservices / Flink)
1. **Consumer Lag (Задержка потребителя):** *(Critical)* 
   *Описание:* Разница между последним сообщением в партиции и offset-ом, который прочитал консьюмер.
   *Влияние:* Если Driver Service отстает от топика заказов (Lag растет), это означает, что время назначения машины для пассажира стремительно увеличивается.
   *Действие:* Auto-Scaling (увеличение реплик подов консьюмера).
2. **Message Processing Rate (Событий обработано в сек):** 
   Показывает фактическую пропускную способность микросервиса.
3. **Dead Letter Queue (DLQ) Size (Размер очереди мертвых писем):** *(Critical)* 
   *Описание:* Отражает число сообщений, упавших при десериализации или фатальных бизнес-ошибках.
   *Влияние:* Рост означает "Poison Pill" — сбой в контрактах (невалидные JSON/Avro), приводящий к потере бизнес-данных. Требует ручного вмешательства.
4. **End-to-End Latency по критическим путям:**
   Замер времени от появления `RideRequested` до издания `PaymentHoldCompleted`. Измеряется через спаны Jaeger. Помогает следить за соблюдением SLA из ADR-001.

### 3. Интеграция в C2 Диаграмму
Инструментарий будет явно выведен на целевую C2 диаграмму событийной архитектуры:
- **Prometheus** стягивает показатели (Pull) от микросервисов, Kafka и Flink.
- Консьюмеры подключены через Kafka Brokers, а их статус и Trace-сборщики (OpenTelemetry) отображены системно.

## Компромиссы
- **Затраты на хранилище данных (Storage Cost).** Распределенная трассировка генерирует чудовищные объемы метрик. Будет внедрено **Probabilistic Sampling**: мы будем сохранять полные трейсы только для 1-5% успешных запросов, но для 100% запросов с ошибками (Error Rate).
- **Синхронизация времени во всех дата-центрах.** EDA метрики и Distributed Tracing теряют смысл, если часы (NTP) на машинах разъезжаются хотя бы на пару секунд.
