# ADR-011: Пайплайн данных и интеграция ML платформы

## Статус
Принято

## Контекст
Для обеспечения ключевых бизнес-функций («GoFuture» динамическое ценообразование, прогнозирование спроса, антифрод) платформе требуется единая стратегия сбора данных. Мы не можем пускать Data Science специалистов и аналитиков напрямую писать SQL-запросы в микросервисные базы:
1. Это вызовет деградацию производительности СУБД из-за тяжелых агрегаций.
2. Данные фрагментированы на 9 изолированных PostgreSQL кластеров (плюс шардированных по 4 гео-регионам). Необходим механизм консолидированного извлечения.

## Решение

Принята **Lambda/Kappa-гибридная архитектура** аналитической платформы, где центром потока данных (Нервной Системой) остается кластер Apache Kafka. Архитектура делится на 4 логических слоя.

### 1. Уровень Сбора (Ingestion Layer)
- Все транзакционные данные из микросервисов выгружаются через механизм **CDC (Debezium + Transactional Outbox)**, зафиксированный в ADR-005. Это не создает нагрузки на master-базы приложения.
- Высоковолатильные потоки (Координаты водителей) сыпятся в Kafka напрямую. 

### 2. Уровень Хранения (Storage Layer - Data Lake & DWH)
- **Data Lake (Сырое хранилище):** Инструмент *Kafka Connect S3 Sink* непрерывно копирует топики в "Облако" (глубокое хранилище типа S3 / HDFS). Там формируется исторический архив в высокосжатом формате **Apache Parquet**. Здесь лежат петабайты сырой, никем не тронутой истории. Это "песочница" для Data-сайентистов.
- **Data Warehouse / DWH (Хранилище очищенных данных):** Выстроенные витрины (Data Marts), оптимизированные для бизнес-отчетов, хранятся в колоночной аналитической СУБД **ClickHouse** (способна читать миллиарды строк за секунды для BI-дашбордов).

### 3. Обработка, Трансформация и ML-обучение (Compute & ML Layer)
- **Real-Time ML (Streaming Analytics):** **Apache Flink** на лету прослушивает Kafka. Он собирает online-фичи (features), например "Плотность водителей в квадрате 2x2 км за последние 5 минут", передает их в Model API "на лету" и скидывает результат расчета (Surge Multiplier) обратно в Kafka.
- **Batch Processing & ETL:** Кластер **Apache Spark** читает паркет-файлы из S3, проводит чистку (убирает null координаты, дубликаты), джоинит таблицы и регулярными батчами перекладывает в ClickHouse каждую ночь.
- **ML Platform (Тренировка Моделей):** В Jupyter / AWS SageMaker Data Scientist-ы обучают новые версии ML моделей (XGBoost/Neural Networks) **исходя из огромной истории S3 Data Lake**.
- Обученные модели деплоятся (Model Serving) в виде независимых микросервисов (или через *MLflow / Seldon*). Транзакционные сервисы (Fraud Service) вызывают эту модель по `gRPC` синхронно в момент заказа поездки.

### 4. Визуализация (BI Layer)
- Для продуктовых менеджеров, отдела финансов и регионального руководства разворачивается платформа **Apache Superset** (альтернативно Tableau или Redash). Этот инструмент подключается непосредственно к ClickHouse и строит дашборды (выручка, отказы, время ожидания такси).

## Компромиссы
- Поддержка отдельного озера S3, СУБД ClickHouse, фреймворков Spark + Flink требует **полноценного штата Data Engineering и MLOps специалистов**.
- Lambda Архитектура означает дублирование части бизнес-логики: код для перевода сырых статусов в бизнес-метрики приходится поддерживать одновременно в кодобазе Stream-обработчика (Flink) и Batch-обработчика (Spark).
