# ADR-013: Дизайн мультитенантной платформы и Onboarding

## Статус
Принято

## Контекст
Стратегия развития GoFuture подразумевает запуск White-Label B2B направления: систему как SaaS будут использовать сторонние таксопарки в новых регионах под собственными брендами. Этим франчайзи-партнерам требуется гарантия изоляции данных, кастомизация (например, свои локальные функции оплаты), а нам — удобный мониторинг и биллинг.

## Решение

### 1. Модель изоляции данных (Data Isolation Model)
Рассмотрено 3 традиционных подхода (Pool, Bridge, Silo).
Для выполнения бизнес-требования "полная изоляция данных и кастомизации" выбран компромисс **Bridge Model: Shared Database, Separate Schema**. 
- СУБД (PostgreSQL кластер) остается общим для снижения инфраструктурных расходов, однако под каждого партнера создается собственная независимая схема базы (например, `schema_partner_x`, `schema_partner_y`).
- **Data Security:** На уровне микросервисов (Go/Python) динамически настраивается `DataSource / Query Routing`. Роутинг извлекает `tenant_id` из токена пользователя и принудительно устанавливает `search_path` в нужную схему СУБД до выполнения запроса. SQL-инъекция со стороны одного партнера физически не сможет затронуть таблицы в схеме конкурента.

### 2. Процесс подключения нового партнера (Automated Onboarding)
Процесс создания партнера оркестрируется новым бизнес-слоем (`Tenant Management Service`).
1. B2B-менеджер GoFuture отправляет POST запрос в **Tenant Management Service**. Задается название партнера, регион и тариф.
2. В IAM-системе (Keycloak) программно по API создается изолированный `Realm` или клиентская группа. Автоматически генерируется профиль **Tenant Admin** с одноразовым паролем.
3. Tenant Service публикует Kafka-событие `platform.tenant.created.v1`.
4. Специальные Database Provisioning Workers, слушающие этот топик, идут в базы каждого микросервиса (Booking, Driver) и выполняют скрипты миграций (создают пустые таблицы в новой схеме для партнера `tenant_x`).
5. Как только все базы рапортуют об успешной настройке схемы, статус тенанта меняется на `READY`, и партнеру улетает письмо на онбординг. Никаких ручных тикетов для DevOps команд.

### 3. Мониторинг мультитенантного окружения (Observability)
Чтобы предотвратить ситуацию "Один партнёр запустил рекламный промокод и положил сервер соседям" (проблема Noisy Neighbor), мониторинг перестраивается:
- **Prometheus Metrics:** В метриках HTTP-throughput микросервисов добавляется label `tenant_id`. Grafana строится так, чтобы инженер техподдержки мог визуализировать график `Error Rate` для конкретного таксопарка.
- **Распределенная трассировка (Jaeger):** Все Span-ы помечаются кастомным тегом `tenant_id`. При медленной работе платформы можно вычислить, затрагивает ли это всех, или только логику базы данных одного партнера.
- **Ограничение (Rate Limiting/Quotas):** На Nginx API Gateway настраивается ограничение нагрузки по лимитам (RPM), прописанным в абонентском договоре конкретного партнера (извлекая ID из JWT Payload).

## Компромиссы
- Подход "Shared DB, Separate Schema" усложняет автоматизацию миграций БД (CI/CD / Flyway). Если платформа обновила код микросервиса Booking, CI/CD скрипт должен прогнать SQL-миграцию `ALTER TABLE` по всем сотням схем партнеров. Процесс релизов становится медленным.
- Введение Tenant Service и Provisioning логики добавляет еще одну оркестрационную систему, критически важную для бизнес-процессов B2B.
