# ADR-014: Интеграция IAM-системы (SSO и Ролевая модель)

## Статус
Принято

## Контекст
Вывод системы в B2B (работа с таксопарками-партнерами) требует полноценного Identity and Access Management (IAM). Партнеры должны авторизовываться своими владельцами и операторами. Крупным партнерам потребуется поддержка корпоративного SSO. Кроме того, строгая ролевая модель (RBAC) обязана защитить персональные данные приложения от несанкционированных действий сотрудников как со стороны платформы, так и внутри самого таксопарка.

## Решение

### 1. Выбор IAM Провайдера (Keycloak)
В качестве централизованного поставщика идентификации разворачивается платформа **Keycloak** (или управляемый SaaS провайдер типа Okta/Auth0).
- **Аутентификация (OIDC / OAuth 2.0):** Микросервисы (Go/Python) избавляются от логики проверки паролей и хранения хэшей. Любой логин пользователя, водителя или диспетчера перенаправляется на форму Keycloak (которая может перенаправить в их корпоративный AD/LDAP - SSO).
- **Структура Токена (JWT):** При успешном логине Keycloak выдает безопасный JWT Access Token. В защищенный (Signed) payload токена зашиваются `user_id`, `tenant_id` партнера и список привязанных `roles`.
- **Авторизация (API Gateway):** Nginx API Gateway математически валидирует подпись JWT (JWKS) на самом краю сети. Если токен подделан или просрочен, запрос сбрасывается (401 Unauthorized), даже не достигнув микросервиса, предотвращая DDoS на бекэнды. 

### 2. Матрица Ролевой модели (RBAC) и Доступов

Ролевая модель разделена на две плоскости: **Platform-level** (сотрудники самого GoFuture) и **Tenant-level** (сотрудники White-label партнера).

*Критическое правило безопасности:* У всех ролей уровня Tenant доступ в коде микросервиса или на СУБД (через Schema/Row Security) жестко органичен **исключительно их собственным `tenant_id` из JWT токена**.

| Роль (Role)            | Уровень Владения    | Зона ответственности (Permissions & Access Level)                                                                                                                                                 | Изоляция данных                                                                                     |
|------------------------|---------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------|
| **Global Admin**       | Platform (GoFuture) | Суперадминистратор платформы. Мутация тенантов в `Tenant Mgmt Service`. Глобальный технический мониторинг (Grafana) инфраструктуры (Kafka/БД). Установка лимитов потребления API для таксопарков. | Неограниченный, видит управление изолированными схемами.                                            |
| **Platform Analyst**   | Platform (GoFuture) | Сводная глобальная финансовая аналитика и Data Science трейнинг.                                                                                                                                  | Имеет доступ к агрегации данных из S3 (без явных имен), видит общую прибыль по всем SaaS партнерам. |
| **Tenant Admin**       | Tenant (Partner)    | Владелец франшизы таксопарка. Настройка интерфейсов, тарифов, инвайты для своих диспетчеров (настройка своей под-группы в Keycloak). Просмотр дашборда выплат водителю.                           | Полный R/W CRUD, но **строго** в рамках своего `tenant_id`.                                         |
| **Tenant Dispatcher**  | Tenant (Partner)    | Диспетчер регионального парка. Создает заказы для звонящих людей, просматривает живую карту своих машин в `Driver Service`, отменяет заказы.                                                      | R/W доступ, ограниченный своим `tenant_id`. Физически не видит машины конкурента по платформе.      |
| **Passenger / Driver** | Tenant (Partner)    | Конечный клиентский профиль мобильного аппа. Создает поездки, принимает поездки.                                                                                                                  | Ограничен только своими собственными активностями (`user_id` + `tenant_id`).                        |

## Компромиссы
- Централизованный IAM (Keycloak) становится главной единой точкой отказа (SPOF - Single Point of Failure) всей глобальной системы. Для SLA 99.99% Keycloak должен разворачиваться в High-Availability режиме (HA) с синхронизацией сессий через Infinispan / Redis поверх нескольких Availability Zones (АЗ) и быть прикрытым кэшом (CDN), чтобы выдерживать Login-штормы.
